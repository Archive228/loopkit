## 30-second quickstart

```bash
# 1. drop the harness into your project (safe: skips existing files)
curl -fsSL https://raw.githubusercontent.com/Archive228/loopkit/main/install.sh | bash

# 2. open Claude Code / Cursor / Codex in the same directory
claude

# 3. write a one-line spec, then let the loop drive
echo "STATUS: not-started" > IMPLEMENTATION_PLAN.md
./run.sh
```

That's it. Skills load only when a task triggers them. The verifier subagent closes the loop.

> Existing files are kept by default. Pass `FORCE=1` to overwrite, or `BACKUP=1` to snapshot your current `.claude/` to `.claude.bak-<timestamp>/` before writing.

## Cherry-pick a single skill

Prefer one skill over the whole floor?

```bash
npx -p claude-loopkit loopkit-add adversarial-verify
```

Installs only that skill into `./.claude/skills/`. Pass `--force` to overwrite an existing copy. Browse the catalog at [skills/](./skills).

## What loopkit is (and isn't)

Straight from my own `.claude/` directory. This is the loadout I actually reach for вҖ” not a methodology, not a lifecycle, not a framework.

**loopkit is:**
- A working harness: settings, hooks, verifier subagent, loop runner. Files on disk, no runtime.
- 49 small skills that load only when relevant, so the agent specializes instead of guessing.
- Cross-agent. Every skill is a plain markdown doc with a YAML header вҖ” nothing Claude-specific inside the skill body.

**loopkit isn't:**
- A new methodology to learn. No BMAD, no Prime Radiant, no 6-phase lifecycle. If a skill doesn't help the loop advance, it isn't here.
- A wrapper CLI. No daemon, no server, no runtime state. `run.sh` is 8 lines.
- A vendor lock-in. Fork it, gut it, keep the three skills you like. MIT.

## The loop

Every skill fires inside this shape. The harness enforces it.

```
  PLAN в”Җв”Җв–¶  ACT  в”Җв”Җв–¶  VERIFY
    в–І                    в”Ӯ
    в””в”Җв”Җв”Җв”Җв”Җв”Җ revise в”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

- **PLAN** вҖ” `spec-first`, `context-budget`, `tool-restraint`, `planner-spec-expand`, `sprint-contract` load here.
- **ACT** вҖ” domain skills load here (debug, security, testing, refactor, docs, data, git-ops).
- **VERIFY** вҖ” `adversarial-verify` + `evaluator-calibration` + the verifier subagent close the loop before anything ships.

Full theory: **[Loop and Harness engineering: 7 files, 5 steps](./docs/effective-harnesses-v03.md)**.

## File-by-file: what you get

```
your-project/
в”ңв”Җв”Җ .claude/
в”Ӯ   в”ңв”Җв”Җ CLAUDE.md          # 60-line standing context (edit for your stack)
в”Ӯ   в”ңв”Җв”Җ settings.json      # permission allowlist + format-on-write hook
в”Ӯ   в”ңв”Җв”Җ agents/
в”Ӯ   в”Ӯ   в””в”Җв”Җ verifier.md    # adversarial verifier subagent (Haiku, JSON-out)
в”Ӯ   в”ңв”Җв”Җ hooks/             # session-start bootstrap
в”Ӯ   в””в”Җв”Җ skills/            # 49 skills, symlinked or copied
в”ңв”Җв”Җ .mcp.json              # MCP server wiring (github, context7)
в”ңв”Җв”Җ MEMORY.md              # cross-session memory index (keep terse)
в””в”Җв”Җ run.sh                 # Plan вҶ’ Act вҶ’ Verify loop runner
```

Every file below is opinionated on purpose. Change what doesn't fit; the loop still works.

### `.claude/CLAUDE.md` вҖ” standing context
60 lines, no more. The tax on every turn. Stack, layout, commands, conventions, the three things the agent must never do. Prune weekly.

### `.claude/settings.json` вҖ” permissions + hooks
Allowlist for read-only Bash and Read; deny-list for `rm -rf` and force-push. `PostToolUse` hook runs Prettier on every Edit/Write so the diff stays reviewable.

### `.claude/agents/verifier.md` вҖ” the adversarial subagent
Runs on Haiku. Reads the diff assuming it's broken. Checks the 11 "fake done" shortcuts (see `skills/adversarial-verify`). Returns JSON. Does not propose fixes, does not run code, does not be polite. This is the load-bearing piece.

### `.claude/hooks/` вҖ” session-start bootstrap
Fires when the agent opens the project. Reminds it of the loop shape and points at `CLAUDE.md`. Zero runtime cost.

### `.claude/skills/` вҖ” the 49-skill library
Each skill: YAML frontmatter with `name` + `description`, a short body, no runtime dependencies. Loads only when its trigger fires. See the full list below.

### `.mcp.json` вҖ” MCP wiring
GitHub + Context7 by default. Add your own; the harness doesn't care.

### `MEMORY.md` вҖ” cross-session index
Preferences, decisions, feedback you keep re-applying. Prune every session or it becomes rot.

### `run.sh` вҖ” the loop runner
8 lines. Reads `PROMPT.md` + `IMPLEMENTATION_PLAN.md`, does one step, verifies, loops until `STATUS: done`. Fresh context each turn; state lives on disk.

## The library

**49 skills across 10 tracks.** Each one: name вҶ’ what it does вҶ’ when it fires.

### agent/llm вҖ” how the agent behaves
- **context-budget** вҖ” trim the working set вҶ’ *before large reads or long sessions*
- **spec-first** вҖ” write the contract before code вҶ’ *any new feature or endpoint*
- **tool-restraint** вҖ” pick the smallest tool that fits вҶ’ *avoids Bash-for-everything drift*
- **subagent-fanout** вҖ” parallelize independent probes вҶ’ *research/audit tasks*

### loop & harness вҖ” long-running, multi-session, multi-agent discipline
- **planner-spec-expand** вҖ” 1вҖ“4 sentence brief вҶ’ full ambitious spec with design language and ordered feature list вҶ’ *starting a fresh project or major feature*
- **sprint-contract** вҖ” negotiate "done" as script-decidable predicates before code вҶ’ *entering an implementation sprint with an evaluator in the loop*
- **feature-list-json** вҖ” enumerate every feature as strict JSON, `passes:false`, editable-passes-only вҶ’ *multi-session builds*
- **init-script-contract** вҖ” idempotent `init.sh` + `test.sh`/`stop.sh`/`reset.sh` siblings, under 120s вҶ’ *setting up a repo for multi-session agent work*
- **progress-reading-protocol** вҖ” fixed 6-step session-open ritual (pwd вҶ’ progress вҶ’ git log вҶ’ feature-count вҶ’ init вҶ’ smoke-test) вҶ’ *any session bootstrapping into an existing project*
- **self-eval-bias** вҖ” interrupt confidently-praise-my-own-work drift вҶ’ *when an agent is about to declare success on its own output*
- **evaluator-calibration** вҖ” few-shot the reviewer persona with rubric anchors to keep skepticism from drifting lenient вҶ’ *before a long autonomous run with a reviewer agent*
- **harness-stripping** вҖ” remove one harness component at a time and measure impact, on every model release вҶ’ *when a new model lands, before piling on more scaffolding*

### debug
- **systematic-debugging** вҖ” hypothesis вҶ’ test вҶ’ narrow вҶ’ *any bug you can't one-shot*
- **read-the-trace** вҖ” extract the actual failure from noise вҶ’ *stack traces, CI logs*
- **bisect-regression** вҖ” git-bisect discipline вҶ’ *"it worked yesterday"*

### security
- **owasp-review** вҖ” top-10 pass on a diff вҶ’ *before merging user-facing changes*
- **authz-check** вҖ” verify every route enforces its policy вҶ’ *auth surface changes*
- **input-validation** вҖ” validate at the edge вҶ’ *any handler taking external data*
- **secret-scan** вҖ” catch keys before commit вҶ’ *pre-push, PR review*
- **dependency-audit** вҖ” CVE + license triage вҶ’ *lockfile changes*

### frontend
- **design-system** вҖ” reuse tokens, don't invent them вҶ’ *any UI change*
- **a11y-pass** вҖ” semantics + keyboard + contrast вҶ’ *before shipping a screen*
- **loading-empty-error-states** вҖ” all four states, not just happy path вҶ’ *any async view*

### testing
- **write-failing-test-first** вҖ” red before green вҶ’ *behavior changes, bug fixes*
- **flaky-hunter** вҖ” reproduce, isolate, quarantine вҶ’ *intermittent CI reds*
- **coverage-gaps** вҖ” find behavior with no test вҶ’ *before declaring "done"*
- **contract-test** вҖ” pin the API shape вҶ’ *service boundaries*

### refactor
- **kill-dead-code** вҖ” prove unreachable, then delete вҶ’ *cleanup passes*
- **simplify** вҖ” collapse indirection you don't need вҶ’ *code review, "too clever"*
- **reduce-nesting** вҖ” early returns, guard clauses вҶ’ *reading fatigue*

### docs
- **changelog-from-diff** вҖ” human-readable release notes вҶ’ *tagging a version*
- **decision-record** вҖ” ADR for the "why" вҶ’ *architectural choices*
- **readme-audit** вҖ” check onboarding path from cold вҶ’ *before public share*

### data
- **sql-review** вҖ” indexes, N+1, plan вҶ’ *any non-trivial query*
- **migration-writer** вҖ” reversible, zero-downtime вҶ’ *schema changes*
- **schema-diff** вҖ” compare shapes, catch drift вҶ’ *env sync*

### git/ops
- **clean-commits** вҖ” atomic, message-first вҶ’ *before PR*
- **pr-from-diff** вҖ” summary the reviewer will actually read вҶ’ *opening a PR*
- **rebase-safely** вҖ” no lost commits вҶ’ *history cleanup*
- **revert-surgical** вҖ” undo only the offending change вҶ’ *bad merges*

### review
- **adversarial-verify** вҖ” the 11 shortcuts agents take to fake "done" вҶ’ *before flipping any task to complete*
- **verification-before-completion** вҖ” run the exact command, read the output, then claim вҶ’ *before any "done" claim*

## Slash commands

Three commands ship in `.claude/commands/`. Each loads a small set of skills and drives a fixed protocol вҖ” no hidden state.

- **Spec first: `/spec`** вҖ” write `PROMPT.md` and `IMPLEMENTATION_PLAN.md` before the agent touches code.
- **Adversarial verify: `/verify`** вҖ” assume the diff is broken; run the 11-shortcut checklist and dispatch the verifier subagent.
- **One-shot polish pass: `/polish`** вҖ” batch quality pass over the current diff (simplify вҶ’ reduce-nesting вҶ’ kill-dead-code вҶ’ a11y-pass вҶ’ loading-empty-error-states вҶ’ readme-audit).

## Install

Pick one. All three drop the same files into your current project.

```bash
# via curl (no Node required, recommended)
curl -fsSL https://raw.githubusercontent.com/Archive228/loopkit/main/install.sh | bash

# via skills.sh (works with any agent that reads SKILL.md)
npx skills add Archive228/loopkit

# via git (if you want to fork first)
git clone https://github.com/Archive228/loopkit
cp -r loopkit/.claude your-project/
```

**Options for the curl installer:**

| Env var | Effect |
|---|---|
| `FORCE=1` | Overwrite existing files instead of skipping |
| `BACKUP=1` | Snapshot existing `.claude/` to `.claude.bak-<timestamp>/` before writing |
| `DEST=/path` | Install into a specific directory instead of `$PWD` |
| `LOOPKIT_REF=<branch>` | Install from a specific branch/tag (default `main`) |

The installer prints every write, verifies the `claude` CLI is on your PATH afterward, and exits non-zero if anything critical failed.

## Compatibility

Works with anything that reads `SKILL.md`:

**Claude Code В· Cursor В· Codex В· Gemini CLI В· Windsurf В· GitHub Copilot CLI**

Every skill file is a plain markdown doc with a YAML header. Nothing agent-specific inside the body. If your agent reads `.claude/skills/`, `~/.claude/skills/`, or a `SKILL.md` file, loopkit fits.

The bundled harness (`settings.json`, hooks, verifier subagent) is Claude-Code-shaped. Cursor/Codex/Gemini users can ignore the harness and use just the skills вҖ” or port the three harness files (they're small).

## Positioning

loopkit is deliberately not a methodology. Approaches like BMAD, Spec-Kit, and full lifecycle skill packs try to help by owning the process вҖ” but they take away the control that makes agents useful in your codebase. loopkit gives you the mini-skills and a floor to stand on. Everything else is yours to shape.

The two failure modes long-running agents actually hit (doing too much at once; premature victory) are structural, not skill-shaped. `run.sh` + the verifier subagent are the structural answer. The skills just make each turn cheaper. The new **loop & harness** track (planner-spec-expand, sprint-contract, evaluator-calibration, harness-stripping, and friends) operationalizes the multi-session discipline described in the companion article.

## Contributing

PRs welcome. The bar:

1. **Skills stay small.** One page, one trigger, one purpose. If it needs 200 lines it belongs in `docs/checklists/`.
2. **Every skill has YAML frontmatter** вҖ” `name`, `description`. Nothing else is required; nothing else should be added without a reason.
3. **No new methodologies.** loopkit doesn't ship lifecycles, phases, or ceremonies. Add a skill, not a framework.
4. **Match the existing file when editing.** Read it before you write.

Adding a skill:

```bash
# 1. copy the template (once available in template/SKILL.md)
cp template/SKILL.md skills/my-new-skill/SKILL.md

# 2. write it
$EDITOR skills/my-new-skill/SKILL.md

# 3. validate the frontmatter shape
./scripts/validate-skills

# 4. open a PR
```
